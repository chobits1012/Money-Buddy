# 雲端自動同步機制 (Auto-Sync) 實作規劃書：方案 B

這份文件詳細列出了將目前的「手動覆蓋式同步」升級為「全自動背景合併與同步 (Auto-Sync, 方案 B)」的具體執行步驟。此規劃可用於開啟新的對話，作為接下來的實作藍圖。

---

## 階段一：資料庫結構升級與時間戳導入 (Database Schema & Middleware)
為了讓系統能夠判斷「誰最新」，我們必須為每一筆獨立資產加上唯一識別碼 (ID) 與最後修改時間戳。

1. **更新 Supabase 資料表結構**：
   * 在 `user_backup` 表中，除了儲存整包 JSON 之外，增加一欄 `last_synced_at` (最後同步伺服器時間)，用來加速比對判斷是否需要拉取最新資料。
2. **重構 Zustand Store 的資料結構**：
   * 確保 Store 中的陣列資料 (如 `customAssets`, `stocks`, `crypto`) 每一個 Item 都有 `id` (使用 `crypto.randomUUID()`) 與 `updatedAt` (時間戳)。
   * 如果現有舊資料沒有 `id` 和 `updatedAt`，需寫一個一次性的 Migration 函式自動幫舊資料補齊。

## 階段二：實作核心 Merge 邏輯 (Conflict Resolution Logic)
這部分是整個機制的大腦，負責在發生衝突時進行合併，確保資料不遺失。

1. **建立 `syncMerge` 函式**：
   * 功能：接收「本地狀態」和「雲端狀態」，並根據 `updatedAt` 進行雙向合併。
2. **陣列資料的 Union 邏輯**：
   * 比對本地和雲端的各類資產 (如 `customAssets`)。
   * **新增的資產**：互相新增。
   * **同時修改的資產**：對於相同 `id` 的物件，比較雙方的 `updatedAt`，保留時間較晚的那一份。
3. **單一數值的處理邏輯（如總資金池餘額）**：
   * 對於非陣列狀態（例如美金資金池餘額 `usdBalance`），同樣比對其對屬的 `updatedAt` 或以最後一次變更事件為主。

## 階段三：實作全自動同步管道 (Auto-Sync Pipeline)
讓同步過程隱形化，背景自動處理。

1. **實作 Pull-First 邏輯 (登入時/開啟網頁時)**：
   * 在 `App.tsx` 或 `portfolioStore` 初始化時：當偵測到使用者已登入，**優先發起一次拉取 (Fetch)**。
   * 將拉取回來的雲端資料，透過階段二開發的 `syncMerge` 放進本地 Zustand Store 進行合併，更新畫面。
2. **實作 背景防抖 (Debounce) 上傳邏輯**：
   * 訂閱 Zustand Store 的變化 (`subscribe`)。
   * 當任何與資產相關的狀態發生改變時（例如新增股票、修改餘額），不需使用者按按鈕，背後自動觸發一個「將本地合併結果上推到 Supabase」的非同步請求。
   * 使用 `lodash.debounce`（或 custom hook），確保如果使用者連續輸入數字（例如 1000），不會發送 4 次 API 請求，而是在停止輸入 2 秒後再整合成一次上傳，節省資源。

## 階段四：UI/UX 翻新 (Status Indicator & Settings)
拿掉已經不合時宜的手動按鈕，換上現代化的同步狀態燈號。

1. **替換「備份管理」為「同步設定」**：
   * 保留目前的「登出」按鈕。
   * 移除原有的「資料上傳」與「下載覆蓋」按鈕。
2. **加入雲端同步狀態燈號 (Sync Indicator)**：
   * 在畫面上方「備份管理」的按鈕旁或內部，加入一個有質感的狀態指示燈：
     * 🟢 **綠燈 / 雲朵打勾圖標**：已與雲端同步完成 (All synced)
     * 🟡 **黃燈閃爍 / 旋轉圖標**：背景同步中... (Syncing...)
     * 🔴 **紅燈 / 斷網圖標**：離線，等待連線後同步 (Offline, pending sync)
3. **錯誤與離線處理**：
   * 如果上傳時發生網路斷線或 API 錯誤，不應阻擋使用者操作，只需將燈號切換為「等待同步」，待有網路時再重試。
