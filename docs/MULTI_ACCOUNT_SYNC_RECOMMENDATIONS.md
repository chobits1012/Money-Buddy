# 多裝置同步與多帳號隔離 — 建議報告書

**目的**：評估「任意裝置增刪資料可同步且不遺失」、「登出後換帳號不互相污染」等需求是否可行，並對照現況提出實作方向與疏漏提醒。

**對照現況（摘要）**：

- 本地資料使用單一 persist key（`portfolio-tracker-storage`），**未依使用者 ID 分區**。
- `PortfolioState` **沒有**紀錄「這包資料屬於哪個帳號」的欄位。
- 雲端備份以 Supabase `user_backup.id = auth.uid()` 區隔，**帳號之間在伺服器端可隔離**（需搭配正確 RLS）。
- 登入後會對「目前記憶體／本地還原的狀態」與「雲端」做 `syncMerge`；若本地其實是上一帳號的資料，**會被當成同一專案合併**，有污染風險。
- 背景 `autoUpload` 會在**未先拉取雲端**下整包覆寫，多裝置情境下可能造成資料被蓋掉（與前一問題獨立但疊加）。

---

## 一、你的目標邏輯是否行得通？

**結論：方向正確，且在業界是標準做法**，但必須補齊三塊：**帳號與本地資料的綁定／分區**、**切換帳號時的明確決策**、**上傳前與雲端的合併或版本控管**。

| 需求 | 可行性 | 說明 |
|------|--------|------|
| 多裝置增刪同步且不見 | 可行 | 需「以 id + updatedAt 合併」+「上傳前 pull-merge-push 或樂觀鎖／版本號」避免最後整包覆寫。 |
| A 離線編輯後再登入 A 能對上 | 可行 | 本地必須標記為 A 的資料集；登入 A 時再與雲端合併後上傳。 |
| 登入 B 時不默默把 A 本地蓋到 B 的雲端 | 可行 | **必須**在 `SIGNED_IN` 且「本地所屬帳號 ≠ 目前 session」時攔截，改為使用者選擇，不可直接 `overwriteState` + `upsert`。 |

## 二、你描述流程中需要補強的「疏漏」與邊界情境

### 1. 本地資料「屬於誰」— 必須可判定

僅「登出前上傳」無法涵蓋：**沒網路、或上傳失敗仍繼續登出**時，本地仍握有 A 的完整資料。下次開啟 App 若直接登入 B，系統必須能分辨：

- 目前磁碟上的這包狀態是 **離線時以哪個身分編輯的**（建議：`localDataOwnerId` 或登出前快照綁定 `userId`）。

若無此欄位，**無法安全實作**「換 B 時要問要不要帶過去／要不要用雲端取代」。

### 2. 「一個瀏覽器、一份 localStorage」≠ 多帳號自然隔離

同一瀏覽器若只有一個 persist 槽，**帳號 A 與 B 會共用同一個 JSON**。實務上應至少擇一：

- **依 userId 分 key**：`portfolio-tracker-storage:<userId>`，登入誰就載入誰（登出時把目前狀態寫回該 key）。
- 或 **單一結構內多包**：`{ "userId-xxx": PortfolioState, ... }`（需注意體積與加密範圍）。

否則「換帳號」只能依賴每次登入覆寫，容易與離線編輯衝突。

### 3. 「把 A 的本地匯入 B」不是單純 merge 而已

`syncMerge` 假設兩邊是**同一邏輯資產庫**的兩個副本。把 **A 時期產生的整包狀態**合併進 **B 的雲端**時：

- 若 A、B 都曾各自建立資產，**UUID 理論上不同**，多為「聯集合併」。
- 若使用者刻意複製或匯入導致 **id 重複**，需定義**以 updatedAt 勝出**或阻擋並提示（與現有陣列合併邏輯一致即可，但要列入測試）。

此操作應列為**高風險、需二次確認**（例如輸入「確認匯入」或勾選）。

### 4. 僅「提醒」仍可能誤操作 — 建議補上備份出口

在「用雲端覆蓋本地」或「捨棄本地」前，提供 **JSON 匯出／下載備份**（可加密），避免不可逆遺失。

### 5. 伺服器端隔離

雲端若以 `user_backup.id = auth.uid()` 儲存，**政策上** A、B 資料不應互見；請確認 Supabase **RLS** 僅允許 `auth.uid() = id`。此為防誤用 API，與前端「不誤傳」互補。

### 6. 多裝置同時線上

即使帳號隔離完善，**同一帳號**多裝置仍須避免「未合併就整包 upsert」。建議與帳號隔離分開規劃：**上傳前 pull + merge** 或 **比較 `updated_at` 再決定是否允許寫入**。

## 三、建議採用的產品流程（與你敘述對齊）

### 情境 A：離線或未登入時編輯，之後登入 **同一帳號 A**

1. 本地狀態標記為 `ownerId = A`（或登出前最後一次成功同步的帳號）。
2. 登入成功且 `session.user.id === A`：
   - 拉取雲端 → `syncMerge(本地, 雲端)` → 寫回 → 再安全上傳。
3. 若曾離線多裝置同時改同一筆 id，仍以 **updatedAt 較新者**為主（與現有合併哲學一致）。

### 情境 B：本地是 **A 的資料**，接下來登入 **B**

**禁止**自動 `syncMerge(本地_A, 雲端_B)` 後直接上傳。

建議阻擋同步並顯示明確選項（文案可再潤飾）：

1. **只使用帳號 B 雲端資料**  
   - 以 B 的 `user_backup` 覆蓋本地顯示（或合併策略明確定為「雲端為主」）。  
   - 實作前建議先提供備份匯出。

2. **將目前本地資料匯入帳號 B（合併進 B）**  
   - 執行 `syncMerge(本地, 雲端_B)` 或「本地為主追加」策略（需產品決策），再 upsert 至 B。  
   - 必須二次確認。

3. **取消登入／改登入 A**  
   - 不變更本地與雲端 B。

（選項 1 與 2 對應你說的「要不要去到 B」「要不要從 B 拉下來取代本地」。）

### 情境 C：登出後「未登入」繼續編輯

需定義產品規則（擇一即可，但要寫清楚）：

- **允許**：視為「仍屬於最後一次帳號 A 的離線草稿」，再次登入時走情境 A；若改登入 B，走情境 B。  
- **不允許**：未登入僅能唯讀（實作較簡單但體驗較硬）。

## 四、實作優先順序建議（Roadmap）

| 階段 | 內容 | 與「不遺失／不污染」的關係 |
|------|------|----------------------------|
| P0 | 本地 **依 userId 分槽** + 狀態內 **`localDataOwnerId`**（或等價 metadata） | 解決跨帳號共用同一 persist 的污染根因 |
| P0 | `SIGNED_IN` 時比對 owner 與 session，不符則 **阻擋自動 sync**，顯示帳號切換對話框 | 對齊你要求的「提醒再決定」 |
| P1 | **autoUpload** 改為 pull-merge-push 或版本 gate | 降低多裝置互蓋 |
| P1 | 覆蓋／捨棄本地前的 **匯出備份** | 降低不可逆遺失 |
| P2 | 更細的衝突 UI（同一筆兩端都改） | 進階體驗 |

## 五、總結

- **你的邏輯在產品與技術上都行得通**，且是處理「離線 + 多帳號 + 雲端備份」的合理模型。
- **目前專案尚未具備**「本地與帳號綁定」與「切換帳號時攔截」兩項能力，因此若不做調整，**無法保證**「換 B 不污染」與「資料永不誤刪」。
- 建議將 **帳號隔離（本地分區 + 登入閘門）** 與 **安全上傳管道** 並列為核心，再疊加備份與進階衝突 UI。
